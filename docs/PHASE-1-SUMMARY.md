# 一期阶段性总结

> tech-doc-mcp 从 0 到一期收尾（约 3 天）。本文梳理"做了什么、怎么做的、现在什么状态、还差什么"。
> 搜索融合的完整决策过程见 [`search-fusion-redesign.md`](search-fusion-redesign.md)。

## 一、项目定位

给 AI 编码 Agent 造一份**永远最新、可查、可溯源**的框架文档记忆，解决"大模型知识停在训练截止那刻、调用过时 API"的痛点。本地运行，通过 MCP 协议把 4 个工具暴露给 Claude Code 等 Agent。

**核心边界（一开始就想清楚的）：** 只服务 Agent 不服务人；返回必须是文档原文（可溯源、不用 LLM 转述）；本地可控可复现。

## 二、已交付的能力

完整链路 **爬取 → LLM 清洗 → 分块 → 向量索引 → serve → 混合搜索** 全部打通并实测：

- **5 个文档源已索引**：fastapi(999)、nextjs(2273)、sqlmodel(384)、tailwindcss(975)、shadcn(608 chunks)
- **MCP 服务实测可启动**，4 个工具（`search_docs` / `read_doc` / `list_sources` / `list_docs`）就绪
- **已推送 GitHub**，密钥 / 源文档 / 大文件全部正确排除

## 三、这一期真正的技术含量：搜索

管线是基础设施，**唯一直接产生价值的是"一次搜索返回对的东西"**。这块是重心，也是最有故事的部分：

**1. 一个反直觉的地基决定（测出来的，不是设计的）：向量化时剥掉代码。**
最初普通段落切分 + 向量化，测下来发现代码块混进 embedding 会制造"假相似"、拖累语义。于是形成了干净的分工：**散文 → 语义通道，代码 → 词法通道**。

**2. 剔除 BM25，自研 boost-only 融合。**
根因：API 标识符 89–100% 出现在代码里（`Session` 代码内 2629 次），BM25 的词频加权在代码语料上彻底错配、污染语义。方案：关键词改 **presence + 饱和**、只对语义候选**乘性加权重排**（不引入新页）。参数 `k1=2.0 / λ=0.5 / n=30` 全部**网格搜索 202 配置 + LLM-judge 交叉验证**选出，不是拍的。

**3. 评估并否决了 cross-encoder reranker。**
"业界标准"在我们这个场景（强 bi-encoder + 干净小语料 + 跨语言 + 找唯一权威页 + 高频调用）结构上不划算——边际提升不抵 ~1 万 token/query，且发现 LLM 裁判过度宽松使其优势存疑。

**4. 根治了三个底层 bug：** `chunk_size` 是死参数、短节合并方向、` ```markdown ` 散文围栏 artifact（~37 篇标题被当代码剥掉）。重索引后 tailwind 那批页语义实打实回升。

## 四、沉淀下来的工程方法（比结论更值钱）

- **一套可回归的评测体系** `eval/`（testset + 网格 + LLM-judge + 召回缺口 + 分块工具），把"搜索好不好"从玄学变成能排名的数字。
- **完整决策记录** `docs/search-fusion-redesign.md`，每个选择的数据依据可追溯。
- **一以贯之的习惯：用数据顶回直觉**——reranker "大家都用"但数据说不划算、chunk-size "调半天"但数据说是噪声、围栏修复"聚合没涨"但个案实打实好。

## 五、当前质量水位

- 纯语义基线 nDCG@10 ≈ **0.90**，混合融合 ≈ **0.93**（25 query 测试集）
- 生产库已是根治后的干净数据

## 六、诚实的短板与待办

1. **评测集是最薄一环**：25 条 query、单人 gold、无统计显著性检验——细微差别（"哪个参数更好"）目前分辨不了，这是下一步最高杠杆。
2. **Web 管理界面（D4）** 一期暂缓，占位符状态。
3. **语言对齐字段是"设计了没接线"**：`language` 从没存过值，跨语言全靠 embedding 扛；要做实需重构。
4. **数据可复现性**：nextjs 缺 raw、shadcn 缺 clean，部分源无法就地重清洗。

## 七、一句话总结

**一期把"最难、最容易静默劣化"的搜索质量，用数据打磨到了可用且可回归的水平，并沉淀了一套让后续改动都能被验证的方法论。** 剩下的短板（评测集厚度、Web、语言对齐）都清楚且不阻塞使用。
