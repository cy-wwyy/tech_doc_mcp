# 搜索融合重构:剔除 BM25,改用 presence + 饱和 + boost-only

> 决策记录(ADR)。记录 2026-07-11 一次完整的"发现问题 → 探索 → 用数据验证 → 敲定方案"的过程。
> 写作时网格搜索已完成、LLM-judge 复核与最终落地待办 —— 见文末「状态」。

---

## 1. 背景与问题

上线后实测:**语义搜索效果不错,但关键词搜索很差,且 BM25 融合后反而污染了语义结果**(把不相关的页挤进 top-K)。

当时的搜索架构(`search/hybrid.py`):
- 语义路:ChromaDB 向量查询
- 关键词路:`BM25Okapi`(`rank-bm25`)+ jieba 分词,启动时全量加载全部 chunk 到内存
- 两路加权 RRF 融合(语义 0.7 / 关键词 0.3)

## 2. 探索过程(按时间顺序)

### 2.1 复现污染
对 fastapi 用 `query="如何在依赖中获取当前登录用户"`, `keywords=["Depends","OAuth2"]` 实测:
- 纯语义 top:`get-current-user`、`simple-oauth2`、`classes-as-dependencies` —— 全部相关 ✅
- BM25 关键词:`["depends","oauth2"]`,把 `depends` 当普通词,按词频排序,精度差
- 融合后 top-10 里 **4 条是 BM25 硬塞的**(`reference/dependencies`、`first-steps`、`advanced-dependencies`、`http-basic-auth`),把语义原有的好结果挤出去 —— **污染确认**

### 2.2 关键洞察:两套文本错配(用户提出,数据验证)
用户提出假设:**向量化时排除了代码,但关键词搜的是含代码全文,可能是代码干扰了结果。**

查 `store/models.py` 证实:
- `text` —— 含代码块,存 ChromaDB,是 BM25/关键词的搜索对象
- `text_for_embedding` —— **去掉代码块**的纯正文,只用于生成向量

统计关键词在"代码块 vs 正文"的分布:

| 关键词 | 代码内出现 | 正文出现 | "只在代码里、正文=0"的 chunk |
|---|---|---|---|
| `Depends` | 414 | 15 | 93 命中中 **82** |
| `OAuth2PasswordBearer` | 69 | **0** | 19 命中中 **19** |

→ BM25 对 `Depends` 打分,本质是数"谁代码块里刷得多",与"正文是否讲当前 query"几乎无关。`OAuth2PasswordBearer` 更是**正文一次都不出现的纯代码标识符**。

### 2.3 规律普遍性(全 5 源)
扩展到所有源,规律一致,且**代码占比越高越严重**:

| 源 | 全文代码占比 | 典型 API 名代码占比 | 极端案例 |
|---|---|---|---|
| nextjs | 42% | 93–100% | `getServerSideProps` 正文 0 |
| fastapi | 50% | 94–100% | `OAuth2PasswordBearer` 正文 0 |
| tailwindcss | 59% | 67–92% | `grid-cols` 92% |
| sqlmodel | 63% | 91–96% | `Session` 代码内 **2629** 次 |
| shadcn | 75% | 84–100% | `DropdownMenu` 代码内 **1310** 次 |

### 2.4 重新定性
- 纯代码标识符恰是关键词搜索的**刚需**(语义索引看不到它们)→ 关键词**必须搜代码**
- 但 `DropdownMenu` 刷 1310 次不代表更相关 → 应按 **presence(有没有)** 而非 **TF(几次)** 打分
- `$contains` 精确子串是布尔命中,天然免疫 TF 膨胀。实测 `$contains("OAuth2PasswordBearer")` 精确指向 `first-steps.md`,`$contains("JWT")` 指向 `oauth2-jwt.md` —— 精度远好于 BM25

### 2.5 为什么剔除 BM25(而非调参)
两个**独立**的理由,叠加后 BM25 在本项目结构上就不合适:

| BM25 的组成 | 在本项目 | 原因 |
|---|---|---|
| 分词(tokenize/jieba) | **不需要** | 输入已是 Agent 抽好的精确 `keywords`,不是自由文本 query |
| TF-IDF 统计打分 | **用错了** | 代码符号高频重复 → 分数膨胀 |

BM25 的通用价值在于"对自由文本 query 分词 + 词法召回补语义短板";而本项目把这个职责**上移**给了 Agent 主动传 `keywords`,再用 presence 精确命中。于是 BM25 + `rank-bm25` + `jieba` 整体移除,连启动时全量加载内存索引也一并去掉。

### 2.6 融合定位:boost-only(用数据选出)
原型对比"朴素 RRF(关键词独立召回一路)" vs "boost-only(关键词只对语义候选加权,不引入语义没召回的新页)":
- 统计每个 case 里"满额命中关键词、但语义 top-30 没召回"的页 —— 这是 boost-only 唯一的理论损失
- 结果这些页**几乎全是噪声**(nextjs 的 `redirects/proxy` 只是迁移说明顺带提到;shadcn 的 `table/button` 只是示例代码里用了组件)
- 而真正相关的关键词页,**语义 top-30 里本来就有**
- 本质:一个页面若语义排名低却含某 API 名,通常是"在代码示例里用了它",而非"在讲它"

→ **关键词的价值不在补召回,而在把"既相关、又精确含 API 名"的页顶上去。** boost-only 根除污染,同时保留这个价值。

### 2.7 借鉴 BM25 的饱和(用户提出)
纯 presence(0/1)丢掉了"出现 1 次 vs 3 次"的信息。用户提出:**1 和 3 应该有别,但 3 和 300 没啥差别** —— 这正是 BM25 的 TF 饱和曲线 `sat(f)=f/(f+k1)`。

区分两个频率信号:
- **单关键词 TF**:一个词在一页出现几次 → 用 `sat` 饱和
- **跨关键词覆盖度**:一页命中几个不同关键词 → 越多越好

一个式子统一两者:`kw_score = Σ_kw sat(count(kw)) / n_kw`。例:`Depends×3,OAuth2×1`(0.75+0.50=1.25)自动赢过 `Depends×300,OAuth2×0`(0.997)—— 覆盖两个词 > 刷一个词 300 次。

补充:**k1 越大,低端分辨率越高**(1↔3 拉得更开),而高端 3≈300 照样塌缩。

### 2.8 方法论转向:不再拍脑袋,用数据敲定
加性 vs 乘性、λ、k1、语义归一化 —— 这些不该在对话里主观指定。改为:**建评测集 → 定指标 → 网格搜索 → 用分数排名 → 敲定**。这同时落地 ROADMAP D3「搜索质量验证体系」。

## 3. 已锁定的设计方向

1. **剔除 BM25**(`rank-bm25` + `jieba` 依赖、`search/bm25.py`、启动全量加载)
2. **关键词通道**:presence + 饱和,`kw_score = Σ_kw sat(count(kw), k1)/n_kw`;因 boost-only 只评语义候选,直接在候选 `text` 上 `count`,连 `$contains` 都不需要
3. **融合 = boost-only**:关键词只对语义候选重排,不引入语义未召回的页
4. **具体公式/参数(k1、λ、加性/乘性、归一化、n)**:交给网格搜索定

## 4. 实验:网格搜索

评测集:`eval/testset.yaml`,25 条 query × 5 源(含单/多关键词 + 高代码 TF 对抗样本),人工 gold 标注。
框架:`eval/harness.py`,缓存 embedding + 语义候选后纯内存跑 202 套配置。指标 nDCG@10 / MRR / Recall@5(gold 被噪声挤下 → 指标降,故 nDCG/MRR 即"抗污染"信号)。

**纯语义基线:nDCG@10 = 0.8993**

| 排名 | 配置 | nDCG@10 | MRR | R@5 | vs 基线 |
|---|---|---|---|---|---|
| 1 | **additive · raw · k1=2.0 · λ=0.25 · n30** | **0.9302** | 0.9333 | 0.9800 | **+0.031** |
| … | additive · raw/minmax · k1=1.5–2.0 · λ=0.15–0.5 | ~0.928 | | | +0.028 |
| 中部 | multiplicative · minmax | ~0.925 | | | +0.025 |
| 201–202 | additive · raw · **λ=1.0** | **0.857–0.862** | | | **−0.037~−0.042** |

**从数据读出的规律:**
- **additive + raw 归一化 + 低 λ(0.15–0.25)+ 高 k1(1.5–2.0)** 最优
- λ 是关键:**过高(=1.0)反而低于基线** —— raw 语义分值域小,λ 太大关键词直接压过语义 → 退化成关键词排序 → 污染回归
- `minmax` 归一化把 top1–topN 强行拉成 [1,0],抹平语义真实差距,略逊于 `raw`
- `n=30` 与 `n=60` 对最优配置无差异 → 取 30 更省
- 关键词整体只带来 +0.03 nDCG:因为**语义本已很强**,关键词是锦上添花的精修,不是主力 —— 这与 boost-only 的定位自洽

## 5. 已知边界:boost-only 的召回缺口(实测不咬)

**缺口链条**:关键词只在代码里 → 去代码的 `text_for_embedding` 不含它 → 该 chunk 向量不反映这个词 → 语义可能召不回 → boost-only 只在语义候选里搜 → **这个 chunk 够不着**。

**为什么通常不发生**:决定相关性的不是"正文有没有这个词",而是"**正文讲不讲这个主题**"。代码周围的正文一般会描述代码在干嘛,所以哪怕标识符只在代码里,页面往往因**主题**被语义在页面级召回(两通道在 token 级分家、页面级重叠)。

**实测**(`eval/recall_gap.py`):36 个 gold,**100% 落在语义 top-30 内**,0 个落在 top-60 之外 → **本语料不存在此缺口**。boost-only 安全,`n=30` 已足够;IDF 门控召回**暂不需要**。

**保留**:testset 的 gold 偏"正文较厚的典型页",未专门构造"纯签名 API 桩页"压测最坏情况。若未来出现召回问题,原则性修法是 **IDF 门控召回**(仅当关键词在全库很稀有、只有 2~3 页含它时,才把语义没召回的关键词页补进来;高频词不补以避噪)。

## 6. 不采纳的方案:清洗阶段扩写 `text_for_embedding`(仅记录,不实现)

**想法**:对正文稀薄、以纯代码符号为主题的页,在 LLM 清洗时补一点内容,让向量"看得到"这些符号,从源头堵住 §5 的缺口。

**若要做,必须守的红线**:
1. **只扩写进 `text_for_embedding`,绝不碰返回给 Agent 的 `text`。** 返回的必须永远是文档原文,否则等于把 LLM 生成内容当官方文档返回,provenance 崩坏。
2. **优先"符号提取"而非"自由总结"**:从代码里抽出字面存在的标识符,作为一行附到 embedding 文本尾部(如 `Symbols: OAuth2PasswordBearer, Depends`),零幻觉;避免让 LLM 自由描述代码(有幻觉风险,且易到处生成同款套话、反而制造假相似)。

**决策:本项目不做这件事。** 理由:§5 实测 0 缺口,属锦上添花而非必需;且需全量重清洗 + 重索引,成本不低。此处仅作为**待选思路存档**,不写入代码。

## 7. code-review 加固(落地后)

高强度 code-review 发现并修复:
1. **关键词改词边界匹配**(`hybrid.count_token`):原 `text.count(kw)` 是纯子串,短/常见词(`id`)会命中 `video`/`grid`/`identifier` 内部并被饱和放大误排;改为对词字符边施加 `(?<!\w)/(?!\w)` 断言,兼容 `@apply`/`table=True`/`use server` 等含符号关键词。`eval/harness.py` 复用同一 `count_token`,保证评测与生产一致。
2. **语义归一化加下限** `SEM_FLOOR=0.1`:min-max 会把候选池最差项归一化到 0、乘性后彻底压死(即便精确命中关键词);改映射到 `[0.1, 1]`。
3. **评测缓存改内容寻址**:缓存文件名带 `(source,version,query)` 哈希,改评测集自动失效对应项,消除按下标命名导致的错位。
4. **judge 打分取末位数字**:`re.findall(r"[0-3]")[-1]`,防模型前言里的数字(如 `OAuth2` 的 2)被误解析。

修复后重跑网格 + judge:结论不变,选定配置(乘性 minmax k1=2.0 λ=0.5)仍为 judge 榜首(0.9127 > 基线 0.8889)。
